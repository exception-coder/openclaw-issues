# 2026-03-13 · 小红书发布超时后重复触发，导致重复发帖

## 元信息

- **分类**：channels
- **状态**：已定位 / 已临时修复
- **影响范围**：单用户
- **相关组件**：control-ui / exec / node-service / xiaohongshu-mcp
- **相关问题**：无

## 问题现象

- 用户要求使用 `xiaohongshu-mcp` 技能发布一篇小红书图文
- 发布链路中先后触发了多次 `publish`
- 用户侧实际出现同内容重复发帖 3 次
- 过程里出现过超时与重试：
  - `exec` 首次 `publish` 调用超时
  - 随后又尝试经由 node `system.run` 发布
  - 最后再改成后台 `exec` 发布

## 症状 → 优先排查项 → 修复命令

| 症状 | 优先排查项 | 修复命令/动作 |
|---|---|---|
| 外部平台重复发帖 | 是否在“未知是否成功”的情况下进行了重试 | 不要立即重试；先查询平台结果 / 本地任务状态 |
| `publish` 超时 | 超时发生在“请求未送达”还是“平台已受理但客户端未等到结果” | 将发布改为单次后台任务，并以幂等检查收口 |
| 同一内容连续出现多次 | 是否混用了多条执行链路（exec / node.run / background exec） | 固定单一执行链路；一次发布只允许一个 in-flight 任务 |

## 初步判断

- 首次 `publish` 虽然在调用侧表现为超时，但很可能平台侧已经受理或正在处理。
- 后续在未确认首个发布结果前，又发起了新的发布请求，导致重复创建内容。
- 根因不在小红书账号本身，而在“把超时误判为失败并直接重试”的执行策略。

## 检查过程

### 1) 检查 skill 与客户端能力

- 命令/工具：

```bash
python3 ~/.openclaw/skills/public/xiaohongshu-mcp/scripts/xhs_client.py --help
python3 ~/.openclaw/skills/public/xiaohongshu-mcp/scripts/xhs_client.py status
```

- 结果：本地 `xiaohongshu-mcp` 服务可用，账号已登录。
- 结论：基础发布能力正常，不是“未登录”或“服务未启动”。

### 2) 检查发布动作的执行链路

- 命令/工具：

```bash
python3 ~/.openclaw/skills/public/xiaohongshu-mcp/scripts/xhs_client.py publish ...
```

- 结果：
  - 第一条 `exec publish` 超时
  - 随后又尝试 node `system.run publish`
  - 最后又改为后台 `exec publish`
- 结论：同一篇内容在“结果未知”时被多次提交。

## 根因结论

> 对外部写操作缺少幂等与结果确认：首次发布超时后，未先确认是否已成功落地，就沿不同执行链路重复提交，导致小红书重复发帖。

## 采取的修复

### 临时修复

说明：

- 在当前会话层面明确规则：
  1. 对外部发布类动作（小红书/社媒/消息发送）一旦超时，不直接重试。
  2. 先检查是否已实际落地，再决定是否补发。
  3. 避免在同一内容上混用 `exec`、`nodes.run`、后台任务三条链路并发/串行重提。
- 已向用户认错，并停止继续重试。

### 最终修复

说明：

- 当前未改动 skill 代码，只完成了操作规范与知识库沉淀。
- 后续应在 skill 或调用包装层加入：
  - 发布请求幂等键（若平台支持）
  - 单次发布任务锁
  - 超时后的“结果查询”步骤
  - 明确区分“提交失败”和“等待结果超时”

## 验证结果

- 验证动作：复盘调用序列与用户反馈
- 实际结果：确认重复发帖由重复提交造成，而非平台自行复制
- 是否恢复：部分恢复

## 风险与回滚

- 风险：如果仍沿用当前人工策略，未来在其他对外写操作上仍可能复现类似问题。
- 回滚方式：无代码回滚；需依赖人工停止重试与人工核查结果。

## 后续建议

1. 为外部写操作增加“超时后先查结果”的统一约束。
2. 对社媒发布类 skill 增加单任务锁，避免重复提交。
3. 能查询最近发布记录的 skill，优先实现“提交后核验”。
4. 用户侧若发现重复内容，第一时间停止自动重试并人工确认。

## 可复用经验

### 下次先查什么

1. 这是“真正失败”，还是“客户端等待超时但服务端已处理”？

3. 该外部平台是否有最近记录/状态查询接口可用于去重确认？
